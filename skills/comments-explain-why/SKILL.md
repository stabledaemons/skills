# Comments Explain Why

Write comments that clarify intent, constraints, tradeoffs, or surprising
behavior. Do not use comments to restate obvious code.

## Rules

1. Explain why the code exists, why this approach was chosen, or what constraint
   it protects.
2. Prefer clearer names or simpler code over explanatory comments when possible.
3. Delete stale, misleading, commented-out, or redundant comments.
4. Keep TODO comments specific: name the missing work, trigger, or follow-up
   condition.
5. Match the repository's existing style for doc comments and inline comments.

## Final Check

- Does the comment explain intent, context, or a non-obvious constraint?
- Would clearer code make the comment unnecessary?
- Is the comment still true after the change?
